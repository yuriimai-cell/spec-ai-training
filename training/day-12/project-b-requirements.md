# 要件定義書
## 商談議事録テンプレートジェネレーター　v1.3

- バージョン：v1.3
- 作成日：2026年4月10日
- 対象システム：社内営業支援ツール（Webアプリ）
- 技術スタック：Next.js 14 / Tailwind CSS v3

---

## 1. 背景

### 1.1 現状の課題

- 記録の抜け漏れ・聞き間違い
  - 商談中の会話スピードに記録が追いつかず、重要な合意事項や数字が漏れる
  - 後から記憶を頼りに補完するため、内容の正確性が担保できない
  - 担当者ごとに記録の粒度・質にばらつきが生じている
- アクション・担当者・期限の曖昧さ
  - 誰が・何を・いつまでにやるかが不明瞭なまま議事録が作成される
  - フォローアップが遅延・漏れ、商談のモメンタムが失われる
  - 口頭合意が文書化されないまま次の商談を迎えるケースが多発
- 情報の取捨選択の困難さ
  - 雑談・本題・脱線が混在した会話から必要な情報を選別するのに時間がかかる
  - 何が「決定事項」で何が「参考情報」かが不明確なまま共有される
- 商談フェーズごとのテンプレート不在
  - 初回訪問・提案・クロージングで必要な記録項目が異なるが統一フォームがない
  - フェーズに必要な情報が抜ける、または不要な項目を埋めるムダが発生

### 1.2 解決アプローチ

- ブラウザ上で動作する「商談議事録テンプレートジェネレーター」を開発する
- 商談フェーズに応じた構造化フォームを提供し、必須項目チェック・アクションアイテム管理・LocalStorage自動保存により記録品質を担保する

---

## 2. ゴール

### 2.1 定量目標

- 議事録作成時間：平均30〜40分/件 → 15分以内/件（50%削減）
- 必須項目の記入率：約60%（担当者依存）→ 95%以上（システム強制）
- アクション明確化率（担当者・期限の記載）：約40% → 100%（入力必須化）
- フェーズ別テンプレート数：0種類 → 5種類
- 入力途中のデータ消失件数：把握不能 → 0件（自動保存で防止）

### 2.2 定性目標

- 商談中でもストレスなく記録できる直感的なUIを提供する
- 統一フォーマットで営業ナレッジを蓄積できる
- 議事録の品質が担当者のスキルや経験に依存しない仕組みをつくる
- PDF・Word形式で即時出力し、顧客送付・社内共有に対応する
- ブラウザを誤って閉じてもデータが失われない安心感を提供する

## 3. スコープ

### 3.1 作るもの

- フェーズ別テンプレート：初回訪問・提案・商談交渉・クロージング・フォローアップの5種類。選択するだけで最適な入力フォームに自動切替
- 構造化入力フォーム：セクション別の入力欄を提供。必須フィールドは空欄のまま出力できない仕組み
- アクションアイテム管理：「内容・担当者・期限・優先度」の4項目をセットで入力。複数件の追加・削除に対応
- リアルタイムプレビュー：入力と同時に最終フォーマットを右パネルにLIVE表示
- LocalStorage自動保存：入力のたびにブラウザのLocalStorageへ一時保存。次回アクセス時に自動復元
- PDF出力：ブラウザ印刷機能を利用。@media print CSSで背景色・強調色を適用し清書フォーマットで出力
- Word出力：背景色なし・箇条書き＋表の構成でWord互換ファイル（.doc）をダウンロード。ファイル名は「日付_会社名_フェーズ」で自動生成

### 3.2 作らないもの

- サーバーサイドDB・クラウド保存：LocalStorageで代替。永続管理はv2.0で検討
- ユーザー認証・ログイン：SSOとの連携は別途対応予定
- 音声録音・文字起こし：外部API連携が必要なためv2.0以降
- CRM（Salesforce等）連携：API連携は別プロジェクトで対応
- スマートフォン専用UI：PC優先。モバイル最適化は後続バージョン
- 過去議事録の閲覧・検索：SharePoint / G Driveを活用
- 重要度タグ付け：UIの簡潔さを優先し対象外とする

## 4. ターゲットユーザー

- 対象：営業部 全社員

### 4.1 主要ペルソナ

- ペルソナA：中堅営業担当者（経験3〜7年・新規開拓担当）
  - 状況：週5〜8件の商談をこなし議事録作成に追われている。記録の質にばらつきがあり上長からのフィードバックが多い
  - ニーズ：商談後30分以内に完成・送付できる「埋めるだけ」の仕組みが欲しい
- ペルソナB：新人営業担当者（経験0〜2年・既存顧客担当）
  - 状況：商談中に何を記録すべきかわからず、メモがまとまらない。先輩に添削されることが多い
  - ニーズ：「何を書くべきか」をシステムが教えてくれる構造化された入力画面が欲しい

### 4.2 利用環境

- 利用端末：ノートPC・デスクトップPC（1280px幅以上）
- ブラウザ：Chrome / Edge / Safari 最新版
- ネットワーク：社内LAN / VPN接続環境（外部API通信なし）
- 操作スキル：Webフォームの基本操作ができるレベル
- 利用シーン：商談直後（同日中）または商談中のリアルタイム記録

## 5. 主要機能

### 機能01　フェーズ別テンプレート選択
- 解決する課題：フェーズごとに必要な記録項目が異なるが統一フォームがない
- 5フェーズ（初回訪問・提案・商談交渉・クロージング・フォローアップ）のテンプレートを用意
- フェーズカードをクリックするだけで最適な入力フォームに自動切替
- 入力済み内容はリセット確認ダイアログ後にクリア（誤操作防止）

### 機能02　構造化入力フォーム＋必須項目バリデーション
- 解決する課題：自由記述では記録漏れが発生しやすく担当者のスキルに品質が依存する
- セクションごとに「必須」「任意」を明確に区別し、必須セクションにはバッジ表示
- 出力ボタン押下時に未入力の必須フィールドを赤枠でハイライト＋エラーメッセージ表示
- placeholder（入力例）で各フィールドの記載内容を提示し、新人の記録品質を向上

### 機能03　アクションアイテム専用ブロック
- 解決する課題：誰が・何を・いつまでにやるかが曖昧なままフォローアップが止まる
- 「内容・担当者・期限・優先度（高中低）」の4項目をセットで入力する専用UIを提供
- 「アクションを追加」ボタンで複数件を柔軟に登録・削除可能
- 担当者または期限が空欄のまま出力をブロック
- 出力ドキュメントでは「内容／担当者／期限／優先度」の表形式に整形

### 機能04　LocalStorage自動保存
- 解決する課題：入力途中でブラウザを閉じたり誤操作した場合にデータが消失する
- 保存タイミング：各フィールドへの入力ごとにdebounce処理（500ms）で自動保存
- 保存キー：`meeting_draft_{phase}` の形式でフェーズごとに個別保存
- 復元タイミング：ページロード時に保存データを検出し、復元確認ダイアログを表示
- 保存状態の可視性：画面上部に「最終保存：○分前」インジケーターを常時表示
- 手動クリア：「フォームをリセット」ボタン押下時にLocalStorageも同時クリア
- 容量：1件あたり最大10KB程度、LocalStorage上限（約5MB）に対し実用上問題なし

### 機能05　PDF・Word即時出力
- 解決する課題：議事録作成後に別ツールへ転記する手間が発生しフォーマットが崩れる
- Word出力：背景色なし・箇条書き＋表の構成。Wordで開いてそのまま編集・共有が可能
- PDF出力：ブラウザ印刷機能を利用。@media print CSSで背景色・強調色を適用
- ファイル名は「YYYY-MM-DD_会社名_フェーズ」で自動生成
- 出力前バリデーションにより必須項目が未入力の場合は出力をブロック

## 6. フェーズ別テンプレート定義

### テンプレート①　初回訪問

- 基本情報
  - 商談日時（必須・日付選択）
  - 顧客会社名（必須・テキスト）
  - 先方参加者・役職（必須・テキスト）
  - 自社参加者（必須・テキスト）
  - 商談場所（任意・テキスト）
- 顧客の現状把握
  - 業種・事業内容（必須・テキストエリア）
  - 現在の業務フロー・使用ツール（必須・テキストエリア）
  - 組織体制・意思決定プロセス（任意・テキストエリア）
- 課題ヒアリング
  - 顕在課題・顧客が明言した問題（必須・テキストエリア）
  - 潜在課題・ヒアリングで引き出した問題（任意・テキストエリア）
  - 課題の優先順位・緊急度（任意・テキスト）
- 自社紹介・反応
  - 紹介した内容・資料・デモ（任意・テキストエリア）
  - 顧客の反応・関心ポイント（必須・テキストエリア）
  - 顧客の懸念・ネガティブ反応（任意・テキストエリア）
- 競合情報
  - 比較検討中の競合製品・サービス（任意・テキスト）
- アクションアイテム
  - 内容・担当者・期限・優先度（必須・専用ブロック）
- 次回アポイント
  - 次回日時（必須・日付選択）
  - 次回の議題・目的（必須・テキスト）
  - 送付予定の資料・情報（任意・テキスト）

  - 提示金額・費用構成（必須・テキスト）
  - 導入スケジュール・フェーズ構成（任意・テキストエリア）

  - 顧客の懸念・反論（必須・テキストエリア）

## 7. 画面構成

### 7.1 画面エリア一覧

- アプリヘッダー（固定）
  - アプリ名・説明テキスト・自動保存インジケーター（最終保存：○分前）
  - 常に最上部に固定表示。印刷時は非表示
- フェーズ選択セクション
  - フェーズカード×5（アイコン・名称・説明）
  - カード選択でフォームを動的切替。選択中カードをハイライト
- 入力フォームエリア（左メイン）
  - セクションカード群・入力フィールド・アクションアイテムブロック
  - フェーズに応じて動的生成。縦スクロール
- プレビューパネル（右サイドバー）
  - リアルタイム議事録プレビュー（セクション別）
  - 幅380px固定。入力と連動してLIVE更新
- エクスポートバー（固定フッター）
  - PDF出力ボタン・Word出力ボタン・バリデーションエラーメッセージ・フォームリセットボタン
  - 常に画面下部に固定。出力前にバリデーションを実行

### 7.2 PDF出力とWord出力の仕様

- Word出力
  - 背景色なし（白地）
  - アクションアイテムは標準の表形式（罫線あり）
  - ファイル形式：.doc（Word互換HTML）
  - ファイル名：日付_会社名_フェーズ.doc
- PDF出力
  - @media print CSSで背景色を適用
  - アクションアイテムは標準の表形式（罫線あり）
  - ファイル形式：.pdf（ブラウザ印刷）
  - ファイル名：日付_会社名_フェーズ.pdf

### 7.3 技術構成

- フレームワーク：Next.js 14（App Router）
- スタイリング：Tailwind CSS v3
- 状態管理：React useState / useReducer（外部ライブラリなし）
- データ永続化：LocalStorage（ブラウザ内、サーバー不要）
- 自動保存：debounce 500ms、保存キー：`meeting_draft_{phase}`
- PDF出力：ブラウザ print API + @media print CSS（背景色付き）
- Word出力：Blob（application/msword）ダウンロード（背景色なし）
- デプロイ：Vercel / 社内サーバー（Static Export対応）
- 想定実装工数：1日（約8時間）※ MVP版
